# 项目讨论记录（功能方向）

## 背景与问题
- 目前工具在多数网站上都会出现类似“Title 过长 / Description 过长 / OG 图片不合规”等提示，造成“几乎所有页面都有问题”的错觉。
- 这种体验会让用户觉得产品在“制造压力”，而非提供价值。
- 因此需要调整方向：减少“问题感”，避免把不重要的事情当成必须修的缺陷。

## 关键共识
- 目标不再是“找问题/修问题”，而是“预览与交付”。
- 产品应该极简实用，重点是“展示效果”和“对外沟通效率”，而不是“诊断清单”。
- 不强调“必须修/警告”，避免用户压力。

## 当前逻辑的核心问题
- 规则与 UI 不一致（测量字体/字号与实际预览不同），导致系统性误报。
- 规则表达过于“问题化”，在真实场景下价值感弱。

## 新方向：预览与交付工具
功能聚焦：
- SERP 预览（桌面/移动）
- 分享卡片预览（OG/Twitter）
- 一键导出 PNG / 复制摘要（用于对外沟通）
- 元信息快照展示（Title/Description/OG 等原始字段，纯展示不评判）
- 可选：与上次快照对比（只显示变化）

表达方式：
- 用“展示效果/是否会被截断”的轻提示替代“问题/警告”。
- 不出现“必须修”，不制造压力。

## 下一步建议
1) 先做功能结构改造：从“诊断列表”转为“预览 + 交付 + 元信息快照”。
2) UI 后置优化：在结构清晰后再做视觉极简化。
3) 目前不考虑付费与分层，先以积累用户与反馈为目标。
